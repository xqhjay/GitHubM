# 需求文档

## 1. 应用概述

### 1.1 应用名称
GitHub 管理器

### 1.2 应用描述
一款通过 GitHub API 深度集成的管理器应用,支持 GitHub 账号授权登录和令牌登录两种方式,提供响应式设计,适配手机端和 PC 端,集成 GitHub 网站的所有核心功能,支持完整的创建、删除、编辑等精细化操作。采用专业应用级美学设计,支持深色模式、浅色模式及跟随系统设置,支持主题色自定义,通过右键上下文菜单提供便捷的操作体验。同时集成 GraphQL API 实现高效数据查询和批量操作能力。新增 PR Diff 对比视图、AI 辅助功能和 Actions 实时日志流查看功能。AI 助手支持任务规划与持续自主执行能力、任务工作流进度持久化及步骤失败自动重试机制。新增 AI 自动生成 changelog 并创建 Release 功能。AI 助手支持多种免费且代码能力强的模型选择。新增模型设置弹窗「测试连接」功能和设置页面 Token 用量统计与费用估算功能。代码浏览页深度优化,手机端采用抽屉形式单栏布局,PC端采用文件树双栏布局,提供懒加载、快捷操作、搜索过滤、文件上传、在线编辑等便捷功能。AI 助手聊天页新增指数退避重连策略和工具执行日志持久化与断点恢复能力。代码编辑器采用 CodeMirror 6,提供轻量级、移动端友好的编辑体验。

## 2. 用户与使用场景

### 2.1 目标用户
- GitHub 用户(个人开发者、团队成员、项目管理者)
- 需要便捷管理 GitHub 资源的用户
- 需要高效批量操作和复杂查询的高级用户
- 移动端用户(需要在手机上编辑代码)

### 2.2 核心使用场景
- 个人项目管理与维护
- 团队协作与代码审查
- 仓库监控与数据追踪
- 移动端快速处理 GitHub 事务
- 移动端代码编辑与提交
- 精细化文件与仓库管理操作
- 复杂数据查询与批量操作
- 自动化 Release 发布与 changelog 生成
- AI 助手长时间任务执行与断点恢复

### 3.2 登录页
- 展示应用 Logo
- 提供两种登录方式:
  - GitHub 账号授权登录按钮
  - GitHub 令牌输入框及登录按钮
- GitHub 账号授权登录流程:
  - 点击授权登录按钮后跳转至 GitHub OAuth 授权页面
  - 用户在 GitHub 授权页面确认授权
  - 授权成功后跳转回应用并自动完成登录
  - 保存授权凭证用于后续 API 调用
- 令牌登录流程:
  - 验证令牌有效性并完成登录
  - 保存令牌用于后续 API 调用
- 支持多账号保存(包括授权登录和令牌登录的账号)
- 支持深色模式、浅色模式及跟随系统设置

### 3.3 首页(仪表盘)
- 展示用户基本信息(头像、用户名、简介)
- 显示仓库统计数据(总数、星标数、Fork 数)
- 展示最近活动时间线
- 快速访问入口(常用仓库、待处理 Issue、待审查 PR)
- 展示「我的收藏」卡片,点击跳转至我的收藏页面
- 通过 GraphQL API 聚合展示跨仓库统计数据

### 3.4 我的收藏(Star仓库列表)
- 展示当前用户添加了 Star 的所有仓库列表
- 显示仓库基本信息(仓库名、描述、Stars 数、语言、最后更新时间)
- 仓库名称仅显示仓库名(例如:GitHubM),不显示所有者名称
- 点击仓库项跳转至仓库详情页时,路由参数使用完整的 fullName(例如:qq5855144/GitHubM)
- 支持按名称、更新时间、星标数排序
- 支持搜索收藏的仓库
- 支持取消 Star 操作

### 3.5 仓库管理

#### 3.5.1 仓库列表
- 展示用户所有仓库(包括拥有的和参与的)
- 仓库名称仅显示仓库名(例如:GitHubM),不显示所有者名称
- 点击仓库项跳转至仓库详情页时,路由参数使用完整的 fullName(例如:qq5855144/GitHubM)
- 支持按名称、更新时间、星标数排序
- 支持筛选(公开/私有、语言类型)
- 显示仓库基本信息(名称、描述、语言、星标数、Fork 数)
- 支持搜索仓库
- 支持批量选择仓库进行操作
- 支持右键点击仓库项触发上下文菜单,菜单项包括:
  - 查看详情
  - Star/取消 Star
  - Fork
  - Watch/取消 Watch
  - 删除仓库
  - 创建分支
  - 查看 Actions
  - 查看 Issues
  - 查看 Pull Requests
  - 部署设置
  - 复制仓库链接
  - 在新标签页打开
- 使用 GraphQL API 实现高效的仓库列表加载和筛选

#### 3.5.2 仓库详情
- 展示仓库完整信息(描述、统计数据、主要语言)
- 显示 README 内容
- 根据仓库归属(isOwner)动态显示不同操作按钮:
  - 自己的仓库(isOwner = true):
    - 编辑仓库信息
    - 删除仓库
    - 管理分支
    - 查看 Releases
    - 查看 Actions Artifacts
    - 查看 Forks 列表(显示被谁 Fork 了)
    - 查看仓库设置
    - 隐藏 Star/Unstar 按钮
    - 隐藏 Fork 操作按钮(改为「查看 Forks」)
    - 隐藏或降级 Watch/Unwatch 按钮
  - 他人的仓库(isOwner = false):
    - Star / Unstar
    - Fork(执行复刻操作)
    - Watch / Unwatch
    - 查看 Issues、PR(只读浏览)
    - 隐藏编辑、删除等管理操作
  - 协作者仓库(collaborator 但非 owner):
    - 可创建 Issue/PR
    - 不能删除仓库
    - 显示中间权限操作
- 展示最近提交记录
- 访问 Issue、PR、代码浏览等子功能入口
- 通过 GraphQL API 一次性获取仓库完整信息

#### 3.5.3 创建仓库
- 输入仓库名称、描述
- 选择公开/私有
- 选择是否初始化 README
- 选择 .gitignore 模板
- 选择许可证类型
- 提交创建请求
- 使用 GraphQL Mutation 执行创建操作

#### 3.5.4 编辑仓库信息
- 修改仓库名称
- 修改仓库描述
- 修改仓库可见性(公开/私有)
- 提交修改请求

#### 3.5.5 删除仓库
- 显示仓库完整名称和基本信息
- 要求用户输入仓库完整名称进行二次确认
- 显示删除警告提示(操作不可恢复)
- 确认后执行删除操作
- 删除成功后返回仓库列表

#### 3.5.6 查看Forks列表
- 展示当前仓库被哪些用户 Fork 了
- 显示 Fork 仓库的基本信息(用户名、Fork 时间、Stars 数)
- 支持点击跳转至 Fork 仓库详情页

#### 3.5.7 批量上传文件
- 支持选择本地多个文件
- 支持选择目标仓库和分支
- 支持指定上传路径
- 输入提交信息
- 显示上传进度
- 完成后显示上传结果

### 3.6 Issue 管理

#### 3.6.1 Issue 列表
- 展示指定仓库的所有 Issue
- 支持筛选(开放/关闭、标签、里程碑、负责人)
- 支持排序(创建时间、更新时间、评论数)
- 显示 Issue 基本信息(标题、状态、标签、负责人、评论数)
- 根据权限显示创建新 Issue 入口(isOwner 或有写权限时显示)
- 支持批量关闭 Issue
- 支持删除 Issue
- 使用 GraphQL API 实现高效的 Issue 查询和批量操作

#### 3.6.2 Issue 详情
- 展示 Issue 完整内容(标题、描述、状态)
- 显示所有评论及回复
- 根据权限显示添加评论入口
- 支持修改状态(开放/关闭)
- 支持编辑标签、里程碑、负责人
- 支持删除 Issue
- 支持删除评论
- 通过 GraphQL API 一次性获取 Issue 完整信息及评论
- 页面顶部新增「AI 摘要」按钮:
  - 点击后调用 wenxin-text-generation 技能生成 Issue 简洁摘要(100字以内)
  - 显示生成的摘要内容
- 评论输入框旁新增「AI 建议」按钮:
  - 点击后调用 wenxin-text-generation 技能根据 Issue 上下文生成 3 条回复候选
  - 用户选择候选回复后自动填入评论输入框

### 3.7 Pull Request 管理

#### 3.7.1 PR 列表
- 展示指定仓库的所有 PR
- 支持筛选(开放/关闭/合并、审查状态)
- 支持排序(创建时间、更新时间)
- 显示 PR 基本信息(标题、状态、源分支、目标分支、审查状态)
- 根据权限显示创建新 PR 入口(isOwner 或有写权限时显示)
- 支持批量关闭 PR
- 使用 GraphQL API 实现高效的 PR 查询和批量操作

#### 3.7.2 PR 详情
- 展示 PR 完整信息(标题、描述、状态、分支信息)
- 显示代码变更(文件列表、差异对比)
- 显示所有评论及代码审查意见
- 根据权限显示添加评论和审查意见入口
- 支持合并 PR
- 支持关闭 PR
- 支持删除 PR
- 通过 GraphQL API 一次性获取 PR 完整信息、变更及评论
- 页面顶部新增「AI 摘要」按钮:
  - 点击后调用 wenxin-text-generation 技能生成 PR 简洁摘要(100字以内)
  - 显示生成的摘要内容
- 评论输入框旁新增「AI 建议」按钮:
  - 点击后调用 wenxin-text-generation 技能根据 PR 上下文生成 3 条回复候选
  - 用户选择候选回复后自动填入评论输入框
- 新增「查看 Diff」入口,点击跳转至 PR Diff 对比视图

#### 3.7.3 PR Diff 对比视图
- 展示模式切换:
  - 支持 Unified(上下合并)和 Side-by-Side(左右分屏)两种 Diff 展示模式
  - 移动端默认使用 Unified 模式
  - PC 端默认使用 Side-by-Side 模式
  - 用户可手动切换展示模式
- 代码语法高亮:
  - 新增行显示绿色背景
  - 删除行显示红色背景
  - 代码内容支持语法高亮显示
- 行内评审评论:
  - 支持在具体代码行添加行内评审评论(Review Comment)
  - 点击代码行号或行内「+」按钮触发评论输入框
  - 输入评论内容后提交
  - 已有评论的代码行显示评论标记
  - 点击评论标记可查看和回复评论
- 快速评审操作:
  - 页面顶部或底部提供快速评审操作按钮
  - 支持一键 Approve(批准)
  - 支持一键 Request Changes(请求修改)
  - 支持一键 Comment(仅评论)
  - 点击后提交评审结果

### 3.8 代码浏览(深度优化)

#### 3.8.1 响应式布局
- 手机端(宽度 < 768px):
  - 默认单栏布局,仅显示代码内容区
  - 顶部显示「展开文件树」按钮
  - 点击按钮后从左侧滑出抽屉(Sheet/Drawer),展示完整文件树
  - 抽屉内点击文件后自动关闭抽屉并在代码区显示文件内容
  - 抽屉可通过点击遮罩层或返回按钮关闭
- PC端(宽度 >= 768px):
  - 双栏布局:左侧文件树面板 + 右侧代码内容区
  - 左侧文件树面板可收起/展开,收起后仅显示图标按钮
  - 两栏布局充分利用宽屏空间

#### 3.8.2 文件树功能
- 懒加载:
  - 初始仅加载根目录文件和文件夹
  - 展开文件夹时才加载该文件夹的子内容
  - 减少初始加载时间和 API 请求次数
- 搜索过滤:
  - 文件树顶部显示搜索框
  - 输入关键词后实时过滤文件名
  - 过滤结果高亮显示匹配项
- 快捷操作:
  - 文件树顶部显示根目录快捷操作按钮:新建文件、新建文件夹、上传文件、刷新
  - 文件树节点 Hover 时显示快捷操作按钮(新建文件、新建文件夹)
  - 右键点击文件触发上下文菜单,菜单项包括:
    - 查看文件
    - 编辑文件
    - 下载文件
    - 复制文件路径
    - 复制 Raw 链接
    - 重命名文件
    - 移动文件
    - 删除文件
    - 查看文件历史
  - 右键点击文件夹触发上下文菜单,菜单项包括:
    - 打开文件夹
    - 创建文件
    - 创建子文件夹
    - 上传文件
    - 复制文件夹路径
    - 重命名文件夹
    - 移动文件夹
    - 删除文件夹
    - 下载文件夹
  - 长按文件/文件夹(移动端)触发上下文菜单
- 文件图标:
  - 不同文件类型显示对应图标(文件夹、代码文件、图片、文档、配置文件、Markdown 等)
- 分支切换:
  - 文件树顶部显示分支选择下拉框(Select 组件)
  - 切换分支后自动刷新文件树和代码内容

#### 3.8.3 代码内容区功能
- 文件内容展示:
  - 支持代码高亮显示
  - 自动检测文件编码,正确显示中文等非 ASCII 字符
  - 支持多种编码格式(UTF-8、GBK、GB2312 等)
  - Markdown 文件支持渲染预览和源码查看两种模式
  - 图片文件支持预览(JPG、PNG、GIF、SVG、WebP 等)
- 面包屑导航:
  - 顶部显示当前文件路径的面包屑导航
  - 面包屑路径可点击跳转至对应目录
- 在线编辑:
  - 文本文件支持在线编辑
  - 编辑器支持语法高亮、行号显示、字号调节
  - 编辑器内支持搜索功能(查找/替换)
  - 编辑完成后填写 commit 信息并提交
- 全屏模式:
  - 代码编辑器支持全屏模式
  - 全屏模式下提供退出全屏按钮
  - 全屏模式下保持所有编辑功能可用
- 文件下载:
  - 支持单个文件下载

#### 3.8.4 文件上传功能
- 支持拖拽文件至文件树或代码区上传
- 支持点击上传按钮选择本地文件
- 支持批量上传多个文件
- 上传时选择目标目录和分支
- 填写 commit 信息后提交
- 显示上传进度
- 上传成功后自动刷新文件树

#### 3.8.5 写操作规则
- 所有写操作(创建文件/文件夹、删除、重命名、移动、上传)需填写 Git commit 信息
- 新建文件/文件夹时对话框显示目标目录路径
- 操作成功后自动刷新文件树
- 操作失败时显示具体错误原因

### 3.9 提交历史
- 展示仓库所有提交记录
- 显示提交信息(提交者、时间、消息、变更文件数)
- 支持按分支筛选
- 支持查看单次提交的详细变更
- 使用 GraphQL API 实现高效的提交历史查询

### 3.10 分支管理
- 展示仓库所有分支列表
- 显示分支信息(名称、最后提交时间、领先/落后主分支的提交数)
- 支持创建新分支
- 支持删除分支(需二次确认)
- 支持切换默认分支
- 支持批量删除分支
- 使用 GraphQL API 实现批量分支操作

### 3.11 协作者管理
- 展示仓库所有协作者列表
- 显示协作者信息(用户名、权限级别)
- 支持添加协作者
- 支持修改协作者权限
- 支持移除协作者
- 使用 GraphQL API 实现批量协作者管理

### 3.12 通知中心
- 展示所有未读通知
- 支持按类型筛选(Issue、PR、提及、仓库活动)
- 支持标记为已读

  - 步骤完成后标记为 done,失败标记为 error 并自动尝试修复
- 前端工作流面板:
  - 显示任务计划的步骤列表,每步有状态图标(pending/running/done/error)
  - 实时更新当前执行进度
  - 与现有的工具调用历史面板并列,通过 Tab 切换
- SSE 协议扩展:
  - 新增 plan 事件:{type:\"plan\", steps:[{id,title,desc}]}
  - 新增 step_start 事件:{type:\"step_start\", stepId}
  - 新增 step_end 事件:{type:\"step_end\", stepId, status:\"done\"|\"error\"}
  - 新增 step_retry 事件:{type:\"step_retry\", stepId, retryCount}
  - 已有的 tool_start/tool_end/think_start/think_chunk/think_end/content/heartbeat 保持不变
- DeepSeek 思考过程流式实时显示:
  - DeepSeek 模型的思考内容在流式输出时实时展示
  - 思考内容默认自动折叠,点击可展开查看详情
- 对话侧边栏工具调用历史:
  - 在对话页面侧边栏添加本次任务的工具调用历史记录面板
  - 每一步显示:工具名称、执行耗时(毫秒)、执行状态(成功/失败)
  - 历史记录按执行顺序排列
  - 每次新对话清空历史
- 任务工作流进度持久化:
  - AI 执行任务时生成的任务计划(PLAN)保存到 Supabase 数据库
  - 每个步骤的执行状态(pending/running/done/error)实时保存到 Supabase
  - 记录每步的开始时间、完成时间、重试次数
  - 用户可在 AI 助手历史回溯面板中查看过去执行过的工作流
- 步骤失败自动重试:
  - 当某个步骤执行失败(step_end status=error)时,Edge Function 自动重试该步骤
  - 最多重试 2 次
  - 重试间隔递增:第 1 次重试间隔 1 秒,第 2 次重试间隔 2 秒
  - 重试时向前端发送 step_retry SSE 事件,包含当前重试次数
  - 重试次数和结果保存到 Supabase
- Release 自动化功能:
  - 用户在 AI 聊天界面输入「合并 PR 后自动创建 Release」或「帮我生成 changelog 并发布」等指令
  - AI 助手自动执行完整的 Release 自动化工作流
  - 工作流包含以下步骤:
    - 获取最新 Release 信息(调用 get_latest_release 工具)
    - 获取自上次 Release 以来已合并的 PR 列表(调用 get_merged_prs_since 工具)
    - 分析 PR 标题、body、labels 自动生成结构化 changelog
    - 自动推断下一个版本号(基于 semver 规则)
    - 调用 create_release 工具创建 GitHub Release
  - changelog 生成格式:
    ```
    ## What's Changed
    ### 🚀 新功能
    - feat: xxx (#12 by @author)
    ### 🐛 Bug 修复  
    - fix: xxx (#8 by @author)
    ### 🔧 其他改动
    - chore/docs/ci: xxx (#5)
    ```
  - 版本号推断规则:
    - PR label 含 breaking/major → 主版本 +1
    - PR label 含 feature/feat/enhancement 或标题以 feat: 开头 → 次版本 +1
    - 其余(fix/chore/docs/ci) → 修订版 +1
    - 无上一个 Release 时默认从 v0.1.0 开始
  - Release tag 格式为 vX.Y.Z
- 指数退避重连策略:
  - 为无写操作自动重连(triggerReconnectOrWarn)添加带抖动的指数退避策略
  - 重试延迟计算公式:min(1500×2^n, 30000ms) + 随机抖动(0~500ms)
  - 最大重试 5 次后停止并通过 toast 告知用户
  - 用户发送新消息或流式成功完成时重置重试计数
- 工具执行日志持久化与断点恢复:
  - 新增 Supabase 数据库表 tool_execution_logs,记录每次工具调用
  - 表字段包括:session_id、turn_id、tool_name、status、elapsed_ms、result_json 等
  - 新增 Supabase 数据库表 workflow_snapshots,保存每轮对话的 messages 列表和 toolHistory
  - 每轮对话结束时保存最新快照至 workflow_snapshots 表
  - 断网重连时可从最新快照恢复 messages 和 toolHistory
  - 新增 RunHistoryPanel 侧边面板,展示当前 session 的工具调用历史
  - RunHistoryPanel 按 turn 折叠显示工具调用记录
  - 支持一键从最新快照恢复 toolHistory
  - 页面加载时自动 fetch 最新 snapshot,恢复 toolHistory 状态

### 3.43 工具执行历史面板(RunHistoryPanel)
- 展示当前 session 的所有工具调用历史记录
- 按 turn(对话轮次)折叠显示工具调用记录
- 每条记录显示:工具名称、执行耗时(毫秒)、执行状态(成功/失败)、执行时间
- 支持展开/折叠每个 turn 的详细工具调用列表
- 提供「从最新快照恢复」按钮,点击后从 workflow_snapshots 表加载最新快照并恢复 toolHistory
- 恢复成功后显示提示信息
- 数据从 Supabase 数据库 tool_execution_logs 表加载

## 4. 业务规则与逻辑

### 4.1 登录与认证
- 用户首次访问时跳转至登录页
- 支持两种登录方式:
  - GitHub 账号授权登录:通过 GitHub OAuth 授权流程完成登录
  - 令牌登录:验证 GitHub 令牌有效性后允许访问
- 授权凭证或令牌保存至本地,用于后续 API 调用
- 授权凭证或令牌失效时自动跳转至登录页
- 支持多账号同时保存(包括授权登录和令牌登录的账号),用户可随时切换

### 4.2 数据持久化
- 保存所有账号的授权凭证或 GitHub 令牌
- 保存账号登录方式标识(授权登录或令牌登录)
- 保存用户收藏的仓库列表
- 保存用户自定义的筛选条件和排序偏好
- 保存浏览历史记录
- 保存当前使用的账号标识
- 保存用户选择的主题模式(深色/浅色/跟随系统)
- 保存用户选择的主题色方案
- 保存搜索历史记录
- 保存 GraphQL 查询模板和历史记录
- 保存用户选择的 AI 模型
- 保存各平台的 AI 模型 API Key(Google Gemini、DeepSeek、Qwen、Meta Llama)
- AI 任务工作流数据持久化到 Supabase:
  - 任务计划(PLAN)保存到 Supabase,包含任务 ID、任务名称、创建时间、步骤列表
  - 每个步骤的执行状态实时保存到 Supabase,包含步骤 ID、状态(pending/running/done/error)、开始时间、完成时间、重试次数
  - 工作流完成后保存完成时间和最终状态
- AI 用量统计数据持久化到 localStorage:
  - 每次 AI 对话结束时,Edge Function 返回 usage 信息(prompt_tokens、completion_tokens、total_tokens)
  - 前端记录到 localStorage,按模型平台分组存储
  - 记录时间戳,用于近 30 天数据筛选
  - 超出 30 天的数据自动清理
- 访问统计数据持久化到 localStorage:
  - 每次进入/切换页面时记录访问数据
  - 记录维度:页面访问次数(PV)、独立访客(UV,用设备指纹或 localStorage 标识)、按日期分组
  - 数据存储在 localStorage
- 工具执行日志持久化到 Supabase:
  - 每次工具调用记录保存到 tool_execution_logs 表
  - 记录字段:session_id、turn_id、tool_name、status、elapsed_ms、result_json、created_at
- 工作流快照持久化到 Supabase:
  - 每轮对话结束时保存最新 messages 列表和 toolHistory 至 workflow_snapshots 表
  - 记录字段:session_id、turn_id、messages_json、tool_history_json、created_at

### 4.3 响应式适配
- PC 端:采用多栏布局,充分利用屏幕空间
- 手机端:采用单栏布局,优化触控操作体验,采用精美的专业应用级美学设计
- 自动根据设备屏幕宽度切换布局

### 4.4 API 调用规则
- 所有功能通过 GitHub REST API 和 GraphQL API 实现
- 使用用户提供的授权凭证或令牌进行身份验证
- 处理 API 限流情况,展示剩余请求次数
- 优先使用 GraphQL API 实现复杂查询和批量操作
- REST API 作为补充,用于 GraphQL 暂不支持的功能

### 4.5 GraphQL API 使用规则
- 使用 GraphQL API 实现以下功能的高效查询:
  - 仓库列表及详情的聚合查询
  - Issue 和 PR 的批量查询和操作
  - 跨仓库的统计数据聚合
  - 复杂的搜索查询
  - 组织和团队信息的嵌套查询
  - 依赖关系和安全警报的批量查询
- 使用 GraphQL Mutation 实现批量操作:
  - 批量关闭 Issue
  - 批量删除分支
  - 批量标记通知为已读
  - 批量处理安全警报
- 合理设计 GraphQL 查询,避免过度获取数据
- 使用分页机制处理大量数据

### 4.6 代码编码处理
- 获取文件内容时自动检测编码格式
- 优先尝试 UTF-8 解码
- 若出现乱码,依次尝试 GBK、GB2312、ISO-8859-1 等编码
- 正确显示中文及其他非 ASCII 字符
- 针对 Markdown 文件,在渲染前先进行编码检测和转换,确保内容正确显示

### 4.7 多账号切换
- 切换账号时自动使用对应授权凭证或令牌重新加载数据
- 不同账号的数据独立存储,互不影响
- 切换账号后保持在当前页面,仅刷新数据

### 4.8 批量上传规则
- 单次上传文件数量不超过 100 个
- 所有文件使用同一提交信息
- 上传过程中显示实时进度
- 若部分文件上传失败,显示失败文件列表及原因

### 4.9 网站部署规则
- 仅支持已启用 GitHub Pages 的仓库
- 部署源可选择分支(如 main、gh-pages)和目录(根目录或 /docs)
- 部署完成后自动生成访问地址
- 自定义域名需在仓库设置中配置 CNAME 记录

### 4.10 产物下载规则
- 支持单个文件直接下载
- 批量下载时自动打包为 ZIP 格式
- 下载前显示文件总大小

### 4.11 删除操作规则
- 所有删除操作均需用户二次确认
- 删除仓库时需输入仓库完整名称进行验证
- 删除文件夹时需显示将被删除的文件数量
- 删除操作执行后显示成功提示
- 删除失败时显示具体错误原因

### 4.12 创建操作规则
- 创建文件夹时自动创建 .gitkeep 占位文件(GitHub 不支持空文件夹)
- 创建文件时自动检测文件名合法性
- 创建操作需指定目标分支
- 所有创建操作需提供提交信息

### 4.13 批量操作规则
- 批量删除时显示操作进度
- 批量操作失败时显示失败项列表及原因
- 批量操作支持取消
- 批量操作完成后显示成功和失败的统计信息
- 优先使用 GraphQL Mutation 实现批量操作以提高效率

### 4.14 右键上下文菜单规则
- 针对仓库的右键菜单在仓库列表项上触发
- 针对文件的右键菜单在文件树中的文件项上触发
- 针对文件夹的右键菜单在文件树中的文件夹项上触发
- 针对搜索结果中仓库的右键菜单在搜索结果仓库项上触发
- 菜单项根据当前对象类型和状态动态显示
- 菜单操作执行后自动关闭菜单

### 4.15 主题模式规则
- 支持三种主题模式:深色模式、浅色模式、跟随系统设置
- 跟随系统设置时,自动检测系统主题并应用
- 用户手动切换主题后,保存用户选择
- 主题切换时平滑过渡,无闪烁

### 4.16 文件图标显示规则
- 根据文件扩展名自动匹配对应图标
- 文件夹使用统一的文件夹图标
- 未识别类型的文件使用默认文件图标

### 4.17 图片预览规则
- 支持 JPG、PNG、GIF、SVG、WebP 等常见图片格式
- 点击图片文件时在预览窗口中显示
- 预览窗口支持缩放和关闭操作

### 4.18 代码编辑器规则
- 使用 CodeMirror 6 作为代码编辑器
- 支持语法高亮、行号显示、字号调节、代码自动换行
- 支持代码搜索(查找/替换)
- 支持移动端原生文本选择(长按选择、复制粘贴)
- 支持深色/浅色主题切换,跟随应用主题模式
- 全屏模式下隐藏其他界面元素,仅保留编辑器和必要控制按钮
- 提供明显的退出全屏按钮
- 全屏模式下保持所有编辑功能可用
- 退出全屏后恢复原始布局
- 关闭编辑器时从路由历史栈中移除对应页面条目

### 4.19 文件 Raw 链接规则
- 复制 Raw 链接时自动生成文件的原始内容访问地址
- Raw 链接格式为:https://raw.githubusercontent.com/用户名/仓库名/分支名/文件路径
- 复制成功后显示提示信息

### 4.20 搜索结果仓库右键菜单规则
- 搜索结果中的仓库项具有专用右键菜单
- 菜单项针对搜索场景优化,提供快速访问和操作功能
- 菜单操作不影响搜索结果列表状态

### 4.21 OAuth 授权流程规则
- 授权登录时跳转至 GitHub OAuth 授权页面
- 授权成功后通过回调 URL 返回授权码
- 使用授权码获取访问令牌
- 访问令牌保存至本地用于后续 API 调用
- 访问令牌过期时自动刷新或提示重新授权

### 4.22 组织与团队管理规则
- 仅组织管理员可创建、删除团队和修改组织设置
- 团队维护者可管理团队成员
- 团队权限分为读取、写入、管理员三个级别
- 组织成员可查看所属组织和团队信息

### 4.23 仓库模板规则
- 仅仓库所有者可将仓库设置为模板
- 基于模板创建的仓库继承模板的文件结构和初始内容
- 模板仓库可以是公开或私有

### 4.24 代码空间规则
- 代码空间基于仓库创建
- 支持自定义代码空间配置(机器类型、环境变量等)
- 代码空间闲置超过设定时间后自动停止
- 删除代码空间不影响原仓库内容

### 4.25 安全警报处理规则
- 安全警报按严重程度分为严重、高、中、低四个级别
- 标记为已解决的警报不再显示在未解决列表中
- 忽略的警报可随时恢复显示

### 4.26 Secret 管理规则
- Secret 值创建后不可查看,仅支持更新
- Secret 可用于 Actions 工作流和代码空间
- 组织级 Secret 可被组织内所有仓库访问
- 仓库级 Secret 仅限当前仓库使用

### 4.27 环境变量管理规则
- 环境可设置保护规则(如需审批、限制分支)
- 环境变量仅在指定环境中可用
- 环境保护规则可防止未授权的部署

### 4.28 Webhooks 触发规则
- Webhook 可订阅多种事件类型(如 push、pull request、issue 等)
- Webhook 触发时向指定 URL 发送 POST 请求
- 支持配置 Webhook 密钥用于验证请求来源
- Webhook 触发历史记录保留最近 30 天

### 4.29 赞助者管理规则
- 仅启用 GitHub Sponsors 的用户可查看赞助者信息
- 赞助者信息包括公开赞助者和匿名赞助者
- 支持向赞助者发送感谢消息或更新

### 4.30 活动统计规则
- 统计数据基于 GitHub API 提供的数据生成
- 支持按日、周、月、年查看统计数据
- 统计图表支持交互式查看详细数据

### 4.31 GraphQL 查询工具规则
- 查询编辑器支持语法高亮和自动补全
- 查询结果自动格式化为易读的 JSON 格式
- 常用查询模板可保存并快速调用
- 查询历史记录保留最近 50 条
- Schema 文档提供完整的 API 字段说明
- 查询结果支持导出为 JSON 或 CSV 格式

### 4.32 外部链接处理规则(Android WebView)
- 应用内点击跳转至 GitHub 官方网站(github.com)的链接时,在 WebView 内部打开或以新标签页方式打开
- 用户按返回键时,能够回到应用内部页面,而非直接退出应用
- WebView 需正确处理导航历史栈,确保返回逻辑符合用户预期

### 4.33 仓库归属判断规则
- 仓库详情页加载时,判断当前登录用户是否为仓库所有者(isOwner)
- 判断依据:当前登录用户的用户名与仓库所有者用户名一致
- 若用户是仓库协作者但非所有者,标记为 collaborator
- 根据 isOwner 和 collaborator 状态动态显示不同操作按钮

### 4.34 仓库详情页操作按钮显示规则
- 自己的仓库(isOwner = true):
  - 显示:编辑仓库信息、删除仓库、管理分支、查看 Releases、查看 Actions Artifacts、查看 Forks 列表、查看仓库设置
  - 隐藏:Star/Unstar、Fork 操作按钮(改为「查看 Forks」)、Watch/Unwatch(隐藏或降级)
- 他人的仓库(isOwner = false):
  - 显示:Star/Unstar、Fork(执行复刻操作)、Watch/Unwatch、查看 Issues、查看 PR(只读浏览)
  - 隐藏:编辑仓库信息、删除仓库、管理分支、查看仓库设置
- 协作者仓库(collaborator 但非 owner):
  - 显示:创建 Issue/PR、查看 Issues、查看 PR
  - 隐藏:删除仓库、编辑仓库信息
  - 显示中间权限操作

### 4.35 我的收藏页面规则
- 展示当前登录用户添加了 Star 的所有仓库
- 数据通过 GitHub API 获取用户的 starred repositories
- 支持按名称、更新时间、星标数排序
- 支持搜索收藏的仓库
- 点击仓库项跳转至仓库详情页
- 支持取消 Star 操作,取消后从列表中移除

### 4.36 PR Diff 对比视图规则
- 展示模式:
  - 移动端默认使用 Unified(上下合并)模式
  - PC 端默认使用 Side-by-Side(左右分屏)模式
  - 用户可手动切换展示模式,切换后保存用户偏好
- 代码高亮:
  - 新增行使用绿色背景标识

### 4.49 Release 自动化工作流规则
- 触发条件:
  - 用户在 AI 聊天界面输入「合并 PR 后自动创建 Release」或「帮我生成 changelog 并发布」等指令
  - AI 助手识别到 Release 自动化意图后,自动执行完整工作流
- 工作流步骤:
  - 步骤 1:调用 get_latest_release 工具获取最新 Release 信息(tag、发布时间)
  - 步骤 2:调用 get_merged_prs_since 工具获取自上次 Release 以来已合并的 PR 列表(含 labels、body)
  - 步骤 3:分析 PR 标题、body、labels 自动生成结构化 changelog
  - 步骤 4:根据 PR labels 和标题自动推断下一个版本号(基于 semver 规则)
  - 步骤 5:调用 create_release 工具创建 GitHub Release,tag 格式为 vX.Y.Z
- changelog 生成格式:
  ```
  ## What's Changed
  ### 🚀 新功能
  - feat: xxx (#12 by @author)
  ### 🐛 Bug 修复  
  - fix: xxx (#8 by @author)
  ### 🔧 其他改动
  - chore/docs/ci: xxx (#5)
  ```
- 版本号推断规则:
  - PR label 含 breaking/major → 主版本 +1
  - PR label 含 feature/feat/enhancement 或标题以 feat: 开头 → 次版本 +1
  - 其余(fix/chore/docs/ci) → 修订版 +1
  - 无上一个 Release 时默认从 v0.1.0 开始
- 辅助工具函数:
  - get_merged_prs_since:获取指定日期/tag 之后合并的所有 PR(含 labels、body)
  - get_latest_release:快速获取最新 Release 的 tag、发布时间
- System Prompt 升级:
  - 在 AI 助手的 System Prompt 中新增「Release 自动化工作流」标准流程
  - 明确各步骤的执行顺序和依赖关系
  - 强化 changelog 生成格式规范和版本号推断规则
  - 引导 AI 将工作流步骤与工具调用关联

### 4.64 指数退避重连策略规则
- 适用场景:
  - AI 助手聊天页的无写操作自动重连(triggerReconnectOrWarn)
- 重试延迟计算:
  - 重试延迟 = min(1500×2^n, 30000ms) + 随机抖动(0~500ms)
  - n 为当前重试次数(从 0 开始)
  - 第 1 次重试延迟:1500ms + 抖动
  - 第 2 次重试延迟:3000ms + 抖动
  - 第 3 次重试延迟:6000ms + 抖动
  - 第 4 次重试延迟:12000ms + 抖动
  - 第 5 次重试延迟:24000ms + 抖动
  - 第 6 次及以后:30000ms + 抖动(达到上限)
- 最大重试次数:
  - 最多重试 5 次
  - 重试 5 次后停止重连
  - 通过 toast 告知用户重连失败
- 重试计数重置:
  - 用户发送新消息时重置重试计数
  - 流式成功完成时重置重试计数
- 随机抖动:
  - 抖动范围:0~500ms
  - 避免多个客户端同时重连导致服务器压力
